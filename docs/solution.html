MeetVideoCapture: техническое решение

Приложение захватывает уже отрисованное видео Google Meet из окна браузера на Ubuntu X11, показывает crop в Qt/QML preview и по кнопке пишет его в MKV/H.264.

Qt 6 / C++ / QML X11 window capture GStreamer No browser hacking
Что делаем
  • Находим top-level окно браузера через X11.
  • Захватываем pixels этого окна по XID.
  • Обрезаем область Meet video внутри окна.
  • Показываем только crop и, при необходимости, пишем файл.
Чего принципиально не делаем
  • Не перехватываем внутренний WebRTC MediaStream.
  • Не вытаскиваем VideoTrack из браузера.
  • Не используем tab capture API и Chrome extension.
  • Не используем Selenium/CDP/injection/packet sniffing.

1. Основная идея

Google Meet уже декодирует WebRTC-видео и рисует его в окне браузера. Поэтому backend работает ниже уровня браузера: он берет готовую картинку окна из X11. Для приложения это обычный поток raw video frames, а не WebRTC stream.

Google Meet в браузере ↓ браузер декодирует WebRTC и рисует видео ↓ X11 capture top-level browser window by XID ↓ crop области Meet video относительно окна браузера ↓ Qt/QML preview latest frame ↓ optional MKV/H.264 recording

2. Захват окна

X11WindowFinder читает список top-level окон, получает title, className, XID, geometry и PID. Браузеры сортируются выше остальных, но выбрать можно любое окно. После выбора controller передает XID и geometry в capture settings.

ximagesrc xid=<browserWindowId> use-damage=false show-pointer=false ! video/x-raw,framerate=30/1 ! videocrop left=<l> top=<t> right=<r> bottom=<b> ! videoconvert ! video/x-raw,format=BGRA ! queue leaky=downstream max-size-buffers=1 ! appsink emit-signals=true sync=false max-buffers=1 drop=true

UI хранит crop как x/y/width/height. GStreamer videocrop принимает left/top/right/bottom, поэтому controller пересчитывает значения от размеров выбранного окна:

left = cropX top = cropY right = sourceWidth - cropX - cropWidth bottom = sourceHeight - cropY - cropHeight

3. Preview без накопления задержки

Preview построен по latest-frame strategy: хранится только последний кадр. Если UI не успел отрисовать старый кадр, новый кадр его заменяет. Очереди preview нет, старые кадры не копятся.

  • appsink drop=true, max-buffers=1, sync=false.
  • queue leaky=downstream, max-size-buffers=1.
  • VideoFrameProvider хранит latest QImage и сжимает update-сигналы.
  • VideoPreviewItem отрисовывает кадр через Qt Quick scene graph texture node.

4. Запись

Recording не блокирует UI и не тормозит preview. Во время записи capture frames подаются в VideoRecorder; когда запись выключена, этот feed отключен. Recorder имеет bounded queue и при перегрузе начинает dropping, а не увеличивает latency.

appsrc is-live=true format=time ! queue max-size-buffers=30 leaky=downstream ! videoconvert ! video/x-raw,format=I420 ! x264enc tune=zerolatency speed-preset=ultrafast key-int-max=30 bitrate=2500 ! h264parse ! matroskamux ! filesink location=<meet-recording-YYYYMMDD-HHMMSS.mkv>

Формат по умолчанию - MKV/H.264. MKV выбран потому, что он надежнее для live recording: при аварийном завершении меньше риск получить полностью непригодный файл, чем с MP4.

5. Потоки

UI thread

QML, состояние controller, таймер записи, scene graph scheduling. Долгие X11/GStreamer команды сюда не попадают.

Window refresh worker

X11WindowFinder перечисляет top-level окна в отдельном QThread и возвращает готовый список в UI.

Capture command worker

Start/stop/restart/update crop идут последовательно в отдельном QThread, без блокировки QML.

Capture streaming path

GStreamer streaming threads, appsink callback, FPS stats, black/stale sampling, latest frame update.

Recorder worker

appsrc feeding, encoder/muxer/file sink, EOS/finalize без ожидания в UI thread.

Bounded data flow

Нет unbounded queues. Preview не ждет recorder. При перегрузе лучше drop, чем latency.

6. Fallback

Если XID capture дает черный или stale кадр, можно переключиться на ScreenRegionFallback. Тогда ximagesrc берет видимую область экрана по координатам crop. Это не tab capture и не extension: это захват pixels экрана.

Ограничение fallback: если поверх Meet окажется другое окно, оно тоже попадет в capture.

7. Ограничения MVP

  • Только Ubuntu X11. Wayland/PipeWire portal можно добавить отдельным backend позже.
  • Свернутое или скрытое окно браузера может не рендериться.
  • XID capture зависит от compositor/GPU/browser и иногда может быть black/stale.
  • Crop задается численно относительно окна браузера; визуальный overlay можно добавить позже.

8. Дальнейшее расширение на Windows и macOS

Linux/X11 backend должен оставаться первой реализацией общей архитектуры, а не отдельной Linux-only веткой. Для расширения на Windows и macOS платформенный захват нужно отделить от общей обработки кадров.

Platform capture backend ↓ common VideoFrame / frame abstraction ↓ crop / timestamp / format normalization ↓ common frame pipeline ↓ preview / analysis / recorder

Платформозависимыми остаются перечисление окон, выбор native source id, permission model, реальный capture API и работа с native pixel buffer или texture. Общими остаются frame abstraction, crop settings, timestamps, FPS/statistics, black/stale detection, bounded/latest-frame delivery, recorder interface, file naming и единая модель warnings/errors.

Linux

Текущий X11 путь: XID → GStreamer ximagesrc → crop → common frame. Для Wayland логичное расширение: PipeWire / xdg-desktop-portal → selected window или display stream → common frame.

Windows

Основной путь: HWND / selected window → Windows Graphics Capture → Direct3D texture frames → crop/convert → common frame. Не нужно строить backend на GDI screenshots или screenshot loop.

macOS

Основной путь: SCWindow или display → ScreenCaptureKit → CMSampleBuffer / CVPixelBuffer → crop/convert → common frame. Реализация частично будет Objective-C++.

Recorder

На первом этапе можно оставить общий recorder через GStreamer appsrc → x264enc → matroskamux → MKV/H.264, если GStreamer runtime поставляется вместе с приложением. Позже возможны native recorder backends: Linux → GStreamer, Windows → Media Foundation / hardware encoder, macOS → AVAssetWriter / VideoToolbox.

Ограничения Windows: может потребоваться user consent или picker, protected windows могут не захватываться, minimized window может не обновляться, а GPU/driver/compositor могут влиять на результат. В будущем желательно избегать лишнего GPU→CPU copy, если preview и recorder смогут работать с texture напрямую.

Ограничения macOS: нужен Screen Recording permission, пользователь может запретить доступ, protected content может не захватываться, minimized или hidden windows могут не обновляться.

Главное правило: не смешивать платформенный capture backend с общей логикой обработки и записи. Linux/X11, Windows и macOS должны быть разными реализациями одного capture interface. Тогда Ubuntu X11 MVP остается первой платформенной реализацией общей meeting/window capture architecture, а не Linux-only тупиком.

Итог: MeetVideoCapture не взламывает браузер и не трогает WebRTC. Он захватывает то, что браузер уже отрисовал в X11-окне, обрезает нужную область и передает ее в быстрый preview и recorder.
